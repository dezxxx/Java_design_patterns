# Паттерн Factory

Этот проект демонстрирует паттерн **Abstract Factory** на Java с масштабируемой и расширяемой структурой.

## Идея проекта
Приложение работает с разными брендами гитар через общие абстракции, а не через конкретные реализации.

## Основные компоненты
- Guitar — общий интерфейс продукта
- GuitarFactory — общий интерфейс фабрики
- Concrete Guitars — конкретные реализации гитар
- Concrete Factories — фабрики для создания гитар
- GuitarFactoryProducer — выбирает фабрику по бренду
- GuitarPlayApp — точка входа

## Abstract Factory Pattern
Abstract Factory — это порождающий паттерн, который предоставляет интерфейс для создания семейств связанных объектов без указания их конкретных классов.

### Структура
- Seller — продаёт продукты
- Accountant — считает прибыль и зарплаты
- Director — управляет магазином
- StoreTeamFactory — абстрактная фабрика
- OnlineStoreTeamFactory — конкретная фабрика

### Как работает
Фабрика создаёт полную команду:
- Seller
- Accountant
- Director

Клиент использует фабрику для получения объектов и выполнения действий.

### Преимущества
- Нет прямого создания объектов
- Лёгкая расширяемость
- Чистое разделение ответственности

## Пример
```java
StoreTeamFactory factory = new OnlineStoreTeamFactory();
Seller seller = factory.getSeller();
Accountant accountant = factory.getAccountant();
Director director = factory.getDirector();

seller.sellProduct();
accountant.calculateBalance();
director.manageStore();
```

# Adapter Pattern (Class Adapter)

## Обзор
Этот проект демонстрирует паттерн Adapter с использованием подхода class adapter (наследование).

Цель паттерна — позволить несовместимым интерфейсам работать вместе.

В этом примере клиент ожидает интерфейс `Log`, но существующая функциональность предоставляется классом `Logger`.  
Класс `Adapter` наследуется от `Logger` и реализует `Log`, адаптируя один интерфейс к другому.

## Структура

- Log — целевой интерфейс  
- Logger — адаптируемый класс  
- Adapter — адаптер (наследуется от Logger, реализует Log)  
- RunnerLogger — клиент  

## Как это работает

Клиент работает с интерфейсом `Log`.

`Adapter`:
- наследуется от `Logger`
- реализует `Log`
- переопределяет `log()`
- вызывает `logObject()` из `Logger`

## Роли

- Target: Log  
- Adaptee: Logger  
- Adapter: Adapter  
- Client: RunnerLogger  

## Пример потока выполнения

1. RunnerLogger создаёт Adapter  
2. Клиент вызывает `log()`  
3. Adapter получает вызов  
4. Adapter вызывает `logObject()` из Logger  

## Ключевая идея

log() → logObject()

## Вывод

Паттерн Adapter позволяет повторно использовать существующие классы без их изменения.  
В данной реализации используется наследование (class adapter) для адаптации Logger к интерфейсу Log.

Decorator Pattern
Обзор

Этот проект демонстрирует паттерн Decorator на Java.

Цель — динамически расширять поведение объекта, оборачивая его в другие объекты вместо использования наследования.

Структура

Car car = new SportCar(new SpeedCar(new BasicCar()));

Поток выполнения:

BasicCar предоставляет базовое поведение

SportCar оборачивает SpeedCar и добавляет поведение
Возвращается финальный результат
Почему Decorator
добавление поведения во время выполнения
избегание глубокой иерархии наследования
гибкая комбинация функциональности
Преимущества
гибкий дизайн
легко расширять
композиция вместо наследования
Вывод

Каждая обёртка расширяет поведение шаг за шагом.

Facade Pattern
Обзор

Facade Pattern — это структурный паттерн проектирования, который предоставляет простой интерфейс к сложной подсистеме.

Вместо того чтобы заставлять клиента работать напрямую с множеством классов и понимать порядок их взаимодействия, фасад скрывает эту сложность и предоставляет небольшой набор удобных методов.

В этом проекте фасад выступает как единая точка входа, которая координирует несколько объектов подсистемы и скрывает внутреннюю логику от клиента.

Назначение

Основная цель паттерна Facade:

упростить доступ к сложной системе
скрыть внутреннее взаимодействие классов подсистемы
уменьшить связанность между клиентом и системой
предоставить более чистый и читаемый API
Структура
1. Facade

Центральный класс, который предоставляет простые методы для клиента.

2. Subsystem Classes

Классы, выполняющие реальную работу.

3. Client

Класс, который использует фасад вместо прямого взаимодействия с подсистемой.

Как это работает

Без фасада клиенту пришлось бы:

создавать множество объектов
вызывать методы в правильном порядке
знать детали реализации

С фасадом клиент просто вызывает один метод, а фасад делает всё остальное.

Пример идеи

Если система содержит:

CPU
Memory
HardDrive

клиенту не нужно вручную запускать каждый компонент.

Достаточно вызвать:

computerFacade.startComputer();

## Flyweight Pattern
Обзор

Этот проект демонстрирует паттерн Flyweight на Java на примере команды стартап-разработчиков.

Основная идея — переиспользовать уже созданные объекты, а не создавать новые каждый раз.
Это помогает снизить потребление памяти при работе с большим количеством похожих объектов.

В этом примере разработчики переиспользуются по типу:

BackendStartUpDeveloper
FrontendStartUpDeveloper

Если один и тот же тип разработчика запрашивается несколько раз, фабрика возвращает существующий объект.

Структура проекта
StartUpDeveloper — интерфейс flyweight
BackendStartUpDeveloper — реализация backend
FrontendStartUpDeveloper — реализация frontend
StartUpDeveloperFactory — простая фабрика
StartUpDevFactorySupplier — фабрика через Supplier
StartUpRunner — клиент
Как это работает
Клиент запрашивает разработчика по типу
Фабрика проверяет, существует ли объект
Если да — возвращает его
Если нет — создаёт, сохраняет и возвращает
Один и тот же объект используется повторно
Зачем нужен Flyweight
много одинаковых объектов
нужно экономить память
важно переиспользование
Реализации
1. StartUpDeveloperFactory

Простая реализация через условия.

2. StartUpDevFactorySupplier

Гибкая реализация через Supplier.

Пример вывода
Backend developer is created
Frontend developer is created
Backend developer develops server side of startup
Frontend developer develops client side of startup
Backend developer develops server side of startup
Frontend developer develops client side of startup

# Chain of Responsibility Pattern

## Обзор
Chain of Responsibility — это поведенческий паттерн проектирования, который передаёт запрос по цепочке обработчиков, пока один из них не обработает его.

В этом проекте запрос проходит через несколько уровней поддержки:
- `BasicSupport`
- `AdvancedSupport`
- `ExpertSupport`

Каждый обработчик решает, может ли он обработать запрос или передать его дальше.

## Структура проекта

- `Handler` — абстрактный базовый класс, который хранит ссылку на следующий обработчик и определяет общий контракт обработки.
- `BasicSupport` — обрабатывает базовые запросы.
- `AdvancedSupport` — обрабатывает более сложные запросы.
- `ExpertSupport` — обрабатывает экспертные запросы.
- `SupportRunner` — создаёт цепочку и отправляет запросы.

## Как это работает

1. `SupportRunner` создаёт все обработчики.
2. Цепочка строится в следующем порядке:

   `BasicSupport -> AdvancedSupport -> ExpertSupport`

3. Запрос отправляется первому обработчику.
4. Каждый обработчик проверяет:
    - если может обработать — обрабатывает
    - если нет — передаёт дальше
5. Если никто не обработал — запрос остаётся необработанным.

## Пример

- `"basic"` → обрабатывается `BasicSupport`
- `"advanced"` → `AdvancedSupport`
- `"expert"` → `ExpertSupport`
- `"unknown"` → не обработан

## Преимущества

- снижает связанность
- гибкая обработка
- легко добавлять новые обработчики
- поддерживает динамическую конфигурацию цепочки

## Недостатки

- запрос может остаться необработанным
- сложнее дебажить
- зависит от порядка цепочки

## Вывод

Реализация показывает Chain of Responsibility с абстрактным базовым классом, что уменьшает дублирование кода.

# Command Pattern

## Обзор
Command Pattern — это поведенческий паттерн, который превращает запрос в отдельный объект.

## Назначение
Инкапсулировать действия в объекты и выполнять их через посредника.

## Структура

- `Command` — интерфейс
- `Light` — получатель
- `LightOnCommand`
- `LightOffCommand`
- `RemoteControl` — вызывающий
- `RunnerCommand` — клиент

## Как это работает

1. Клиент создаёт получателя  
2. Создаёт команды  
3. Передаёт их в invoker  
4. Invoker вызывает execute()  
5. Команда делегирует работу  

## Преимущества

- слабая связность  
- расширяемость  
- поддержка undo/redo  

## Пример

- Light включает/выключает свет  
- команды вызывают методы  
- RemoteControl управляет  

## Вывод

Command отделяет отправителя от получателя.
